# AI-OJ 智能算法辅助评测系统最终面试稿

## 1. 项目一句话定位

### 极简版
传统 OJ 判题系统上接了一条 AI 辅助分析链路，并通过 SSE 把分析过程流式返回前端。

### Java 后端版
这是一个基于 Spring Boot 的在线判题项目，支持 Java/C++ 判题，并集成了 AI 辅助分析和 SSE 流式反馈。

### AI 应用后端版
这是一个把传统代码判题和大模型辅助分析结合起来的 AI 应用后端项目，重点做了结构化上下文和流式交互。

### OJ 判题版
这是一个支持 Java/C++ 本地编译运行、判题状态分类，并在失败后支持 AI 辅助分析的轻量 OJ 系统。

### 面试开场版
我做的是一个传统 OJ 判题加 AI 辅助分析的项目，用户提交代码后先走正常判题，再手动触发 AI，用 SSE 流式返回分析建议。

## 2. 简历项目描述最终版

### 1. 保守版

- 基于 Spring Boot 3.3.5、Java 17、MyBatis-Plus、Vue3 + Vite 开发在线评测系统，完成题目管理、代码提交和结果展示闭环，技术栈证据充分，风险低。
- 实现 Java / C++ 本地进程级编译运行判题，支持 Accepted、Compile Error、Runtime Error、Wrong Answer、TLE 等状态分类，核心逻辑在 `JudgeServiceImpl`，但当前不是强隔离沙箱。
- 在 Wrong Answer 场景下返回第一个失败用例的 `failedInput / expectedOutput / actualOutput / failedCaseIndex`，让判题结果不止有状态，还有可解释上下文，证据强。
- 基于 LangChain4j 接入 DashScope / Qwen，提供独立 AI 辅助分析接口，通过 `SseEmitter` 流式返回建议，AI 链路独立，不影响传统判题主链路。
- 为 AI 请求补了结构化上下文、字符级 Prompt 长度保护和 `message / done / error` SSE 协议，前端使用 `fetch + ReadableStream` 解析，当前不是严格 Token 级控制。

### 2. 平衡版

- 面向编程练习场景开发 AI-OJ 系统，传统 OJ 判题负责确定性结果，AI 辅助分析负责失败后解释与修改思路，职责边界清晰，适合讲 Java 后端工程实现。
- 后端通过 `ProcessBuilder + 临时目录` 支持 Java / C++ 真实编译运行，按测试用例给出 Compile Error、Runtime Error、Wrong Answer、TLE、Accepted 等结果，能体现判题主链路能力，但不能夸大成企业级安全沙箱。
- 针对 Wrong Answer 和 Runtime Error 场景，把失败输入、期望输出、实际输出、失败用例序号等上下文结构化返回并传给 AI，提升模型分析的针对性，证据强。
- 基于 LangChain4j 的 `QwenStreamingChatModel` 接入 DashScope / Qwen，结合结构化 Prompt 和字符级长度保护，避免长代码、长错误输出直接无控制拼接。
- AI 接口采用 `SseEmitter`，前端通过 `fetch + ReadableStream + AbortController` 实现流式展示、停止生成、重复请求中止旧请求和页面离开中止请求，交互链路完整，但不是异步判题队列。

### 3. AI 应用后端版

- 在传统 OJ 判题基础上接入 AI 辅助分析链路，把题目标题、题面、语言、判题状态、失败用例上下文、错误输出和用户代码结构化传给模型，体现 AI 应用后端的上下文组织能力。
- 使用 LangChain4j 接入 DashScope / Qwen 流式模型，通过 `SseEmitter` 向前端持续输出分析结果，前端用 `fetch + ReadableStream` 做事件解析和增量展示，具备完整流式交互证据。
- 为 Prompt 增加字段级预算和总长度预算，支持 `wrongCode` 的 head + tail 截断，并在日志中记录截断状态而不打印完整用户代码，体现工程化的输入治理能力。
- 在 SSE 生命周期上补了 `message / done / error` 协议、后端 `completion / timeout / error` 日志、前端 `AbortController` 停止生成和请求代次隔离，提升流式链路稳定性。
- 提供 Dockerfile、docker-compose.yml 和 Vercel 配置，能够支撑本地联调和基础部署演示，但不夸大为完整生产级部署体系。

### 4. 冲刺版

- 实现传统 OJ 判题主链路，支持 Java / C++ 本地编译运行、判题状态分类和失败上下文返回，解决“只给状态不够解释”的问题，技术点真实，证据等级 A，风险低。
- 在 Wrong Answer 场景下结构化返回第一个失败用例的输入、期望输出、实际输出和失败序号，再传给 AI 进行分析，增强模型反馈的针对性，证据等级 A，风险低。
- 基于 LangChain4j 接入 DashScope / Qwen 流式模型，结合 `SseEmitter`、`fetch + ReadableStream` 实现 AI 建议流式返回，证据等级 A，风险低。
- 为 AI 请求补充结构化字段和字符级 Prompt 长度保护，控制长代码、长报错的输入规模并保留有效上下文，证据等级 A，风险中。限制：当前不是严格 Token 级控制。
- 补齐 SSE 生命周期和前端中止控制，支持 `message / done / error`、停止生成、重复请求中止旧请求、页面离开中止请求，证据等级 A，风险中。限制：不能说彻底取消上游模型生成。

## 3. 30 秒 / 1 分钟 / 3 分钟项目介绍稿

### 30 秒版

这个项目是一个传统 OJ 判题加 AI 辅助分析的系统。用户提交 Java 或 C++ 代码后，后端会先真实编译运行并按测试用例判题；如果出现编译错误、运行错误或者答案错误，用户可以手动触发 AI 分析，后端把题面、判题状态和失败用例上下文组织好，再通过 SSE 把建议流式返回前端。

### 1 分钟版

我做的这个项目主要有两条链路。第一条是传统 OJ 判题链路，后端用 Spring Boot 接收代码提交，根据语言把代码写进临时目录，再调用 `javac/java` 或 `g++` 去真实编译运行，最后根据超时、退出码和输出比对返回 Accepted、Compile Error、Runtime Error、Wrong Answer 或 TLE。第二条是 AI 辅助分析链路，用户在判题后可以手动点击 AI 分析，后端基于 LangChain4j 接入 DashScope / Qwen，把题目标题、题面、判题状态、错误信息和第一个失败用例上下文结构化传给模型，再通过 `SseEmitter` 流式返回，前端用 `fetch + ReadableStream` 解析并展示。这个项目比较适合体现 Java 后端的判题实现、进程执行、上下文组织和流式交互能力。

### 3 分钟版

这个项目我主要想解决两个问题。第一个问题是，传统 OJ 只告诉用户“对还是错”，但用户不知道到底错在哪。第二个问题是，大模型可以给解释和思路，但如果上下文太乱、交互太粗，反馈质量也不稳定。

所以我先实现了传统判题链路。前端把 `questionId`、语言和代码发给后端，后端先根据题目 ID 查题目和测试用例，然后在临时目录里把代码落成源码文件。Java 走 `Main.java + javac/java`，C++ 走 `main.cpp + g++/可执行文件`。编译阶段如果超时或者退出码不对，就直接判成 Compile Error；编译成功以后按测试用例逐个执行，运行超时判 TLE，退出码异常判 Runtime Error，输出和标准答案不一致判 Wrong Answer，全部通过才是 Accepted。这个部分能体现我对进程执行、结果分类和判题流程的实现。

在这个基础上，我又接了一条 AI 辅助分析链路，但我没有把它直接塞进判题主链路，而是做成独立接口。这样模型挂了也不会影响判题结果。用户判题失败以后，可以手动点击 AI 分析，前端会把题目标题、题面、语言、当前代码、判题状态、判题提示，以及第一个失败用例的输入、期望输出、实际输出这些结构化信息发给后端。后端用 LangChain4j 接 DashScope / Qwen 流式模型，把这些字段组装成结构化 Prompt。为了避免长代码和长错误输出把请求撑爆，我做了字符级长度保护，每个字段有自己的预算，总 Prompt 也有上限，代码字段还是 head + tail 截断，不是只保留开头。

AI 返回这一块我用了 `SseEmitter`，前端因为要发 POST 请求体，所以没有用 `EventSource`，而是用 `fetch + ReadableStream` 自己解析 `message / done / error` 事件。后面我又补了 SSE 生命周期，后端会记录 `completion / timeout / error` 日志，前端用 `AbortController` 支持停止生成、重复请求中止旧请求、页面离开中止请求，再加一个本地请求序号，避免旧流晚到继续污染当前结果。

部署上，仓库里有 Dockerfile、docker-compose.yml 和 Vercel 配置，能支撑本地联调和基础部署演示。整体上，这个项目最适合强调的不是“我做了多复杂的架构”，而是“我把传统判题、结构化 AI 上下文和流式交互结合起来做成了一个完整闭环”。

## 5. 核心链路逐条讲解

### 1. 用户提交代码到传统判题链路

- 一句话概括：前端提交题目 ID、语言和代码，后端查题目和测试用例后进行真实编译运行并返回判题结果。
- 请求入口：`POST /api/judge`
- 核心类/方法：`JudgeController#judge`、`JudgeServiceImpl#judge`
- 数据如何流转：前端提交 `questionId/language/code` -> 后端查 `QuestionDetailVO` -> 临时目录落盘源码 -> 编译 -> 按测试用例运行 -> 返回 `JudgeResponse`
- 已有代码证据：`JudgeController`、`JudgeServiceImpl`、`QuestionServiceImpl`
- 已验证证据：已验证 Accepted / Compile Error / Wrong Answer / Runtime Error
- 当前限制：同步判题，不是异步队列；不是强隔离沙箱
- 面试推荐说法：我做的是一个真实编译运行的轻量判题闭环，而不是纯模拟判题
- 面试危险说法：我做了企业级安全沙箱和高并发异步判题

### 2. Compile Error / Runtime Error / Wrong Answer / TLE 分类链路

- 一句话概括：判题链路会根据编译结果、退出码、输出比对和超时情况返回不同状态。
- 请求入口：`POST /api/judge`
- 核心类/方法：`JudgeServiceImpl#runTestCases`、`executeCommand`
- 数据如何流转：编译失败直接返回 Compile Error -> 运行超时返回 TLE -> 退出码异常返回 Runtime Error -> 输出不一致返回 Wrong Answer -> 全部通过返回 Accepted
- 已有代码证据：`JudgeStatus`、`JudgeServiceImpl`
- 已验证证据：四类失败和通过都已做运行时验证
- 当前限制：内存限制字段当前没有真正落到进程资源限制
- 面试推荐说法：我把判题失败细分成几种标准状态，便于前端展示和后续 AI 分析复用
- 面试危险说法：我已经做了完整 OJ 资源配额和判题状态机

### 3. 第一个失败用例结构化返回链路

- 一句话概括：Wrong Answer 和部分运行失败场景下，后端会返回第一个失败用例的结构化上下文，而不只是一个“错了”的状态。
- 请求入口：`POST /api/judge`
- 核心类/方法：`JudgeResponse`、`JudgeServiceImpl#runTestCases`
- 数据如何流转：测试用例循环里一旦发现第一个失败点 -> 提取 `failedInput / expectedOutput / actualOutput / failedCaseIndex` -> 写入 `JudgeResponse`
- 已有代码证据：`JudgeResponse` 新增字段、`JudgeServiceImpl` 失败分支
- 已验证证据：Wrong Answer 已真实返回 `1 2 / 3 / -1 / 1`
- 当前限制：只返回第一个失败用例，不返回完整失败列表
- 面试推荐说法：我刻意只返回第一个失败用例，控制返回体规模，同时给 AI 一个最关键的失败上下文
- 面试危险说法：我已经做了完整失败样本聚合和多用例智能分析

### 4. 用户手动触发 AI 辅助分析链路

- 一句话概括：用户在判题后手动触发 AI，后端把结构化判题上下文和代码一起交给模型做分析。
- 请求入口：`POST /api/ai/help`
- 核心类/方法：`AiController#help`、`AiHelpServiceImpl#help`
- 数据如何流转：前端收集题面、代码、判题状态、失败用例上下文 -> 请求 AI 接口 -> 后端组装 Prompt -> LangChain4j 调 Qwen -> 流式返回
- 已有代码证据：`AiHelpRequest`、`AiHelpServiceImpl`
- 已验证证据：Compile Error、Wrong Answer、Runtime Error 场景都做了 AI 接口验证
- 当前限制：当前是手动触发，不是判题失败自动触发
- 面试推荐说法：我把 AI 做成独立辅助链路，既能提升反馈体验，也不会影响传统判题主功能
- 面试危险说法：判题失败后系统会自动异步调度 AI 修复

### 5. SSE message/done/error 流式返回链路

- 一句话概括：后端把模型输出拆成 `message / done / error` 事件，前端按事件类型增量处理。
- 请求入口：`POST /api/ai/help`
- 核心类/方法：`AiHelpServiceImpl#help`、`safeSend`、前端 `parseSseBlock`
- 数据如何流转：模型回调 `onNext` -> 后端发 `message` -> 前端拼接 `aiAdvice`；流结束发 `done`；异常发 `error`
- 已有代码证据：`AiHelpServiceImpl`、`App.vue`
- 已验证证据：已验证 `message`、`done`、参数缺失 `error`
- 当前限制：不能说彻底取消上游模型生成；只能说本端停止继续向关闭连接发送
- 面试推荐说法：我把 SSE 从“单纯流式输出”补成了有协议、有生命周期的流式链路
- 面试危险说法：我实现了完整的流式任务调度和可取消模型执行

### 6. Docker Compose / Vercel 前后端部署链路

- 一句话概括：仓库提供了后端容器化和前端部署配置，用于本地联调和基础演示部署。
- 请求入口：部署配置层，不是业务接口

## 6. 重点技术点深挖

### 1. OJ 判题

为什么要真实编译运行：  
因为只有真实编译运行，才能拿到编译错误、运行异常、超时和真实输出，这些是后续 AI 分析最关键的上下文。

Java / C++ 如何支持：  
后端按语言生成不同源码文件和编译运行命令，Java 走 `Main.java + javac/java`，C++ 走 `main.cpp + g++/可执行文件`。

临时目录有什么作用：  
每次提交都在独立临时目录里落盘源码和编译产物，避免不同提交之间互相污染，也方便执行后统一清理。

编译错误怎么捕获：  
编译阶段看超时和退出码，编译失败就返回 Compile Error，同时把编译输出写进响应体。

运行错误怎么捕获：  
运行阶段看退出码和 stderr，如果退出码非 0，就返回 Runtime Error，并携带错误信息。

Wrong Answer 怎么判断：  
把程序输出和标准输出做标准化后比较，不一致就返回 Wrong Answer。

TLE 怎么判断：  
通过 `process.waitFor(timeout)` 判断是否超时，超时后强制销毁进程并返回 TLE。

当前沙箱边界是什么：  
当前只是本地 `ProcessBuilder + 临时目录` 的轻量执行方案，不是容器或内核级强隔离沙箱。

### 2. 失败用例结构化上下文

为什么只返回第一个失败用例：  
因为目标不是做完整测试报告，而是给前端和 AI 一个最关键、最短路径的失败样本。只返回第一个失败用例，响应体更小，AI 也更容易聚焦。

Wrong Answer 返回哪些字段：  
`failedInput`、`expectedOutput`、`actualOutput`、`failedCaseIndex`

Runtime Error 返回哪些字段：  
尽量返回 `failedInput`、`actualOutput`、`failedCaseIndex`，错误信息放在 `message`

Compile Error 为什么不伪造失败用例：  
因为编译阶段压根没有进入测试用例执行，伪造 `failedInput` 会让链路不真实。

这些字段如何传给 AI：  
前端先把 `JudgeResponse` 保存到 `consoleState`，然后在调用 `/api/ai/help` 时把这些字段一起塞进 `AiHelpRequest`

对 AI 分析质量有什么帮助：  
模型不只是看到“答案错了”，还能看到具体哪组输入失败、预期是多少、实际是多少，更容易定位逻辑错误。

### 3. AI 辅助分析

为什么选 LangChain4j：  
因为它在 Java 后端里接入模型比较顺手，尤其是对 DashScope / Qwen 的流式封装比较直接，适合这个项目体量。

`QwenStreamingChatModel` 如何接入：  
后端通过 builder 配置 API Key、模型名和搜索开关，然后用 `generate(messages, handler)` 做流式生成。

Prompt 里包含哪些结构化字段：  
题目标题、题目描述、语言、判题状态、失败用例序号、判题信息、实际输出、错误输出、期望输出、失败输入、用户代码。

字符级长度保护如何做：  
每个字段有预算，总 Prompt 也有上限，超出就截断；代码字段用 head + tail，不只留开头。

为什么不是严格 Token 级：  
因为当前没有接 tokenizer，也没做模型级 token 计数，只能诚实说是字符级输入预算控制。

AI 失败为什么不影响判题：  
因为 AI 是独立接口，传统判题不依赖模型返回，模型故障只影响辅助分析，不影响判题结果。

### 4. SSE

为什么用 `SseEmitter`：  
因为这个项目是 Spring MVC 技术栈，`SseEmitter` 足够轻量，能快速支撑流式返回。

为什么前端用 `fetch + ReadableStream`，而不是 `EventSource`：  
因为这里需要 POST 请求体，把题面、代码和判题上下文传过去，`EventSource` 更适合 GET。

`message / done / error` 协议怎么设计：  
正常增量输出走 `message`，流结束走 `done`，参数缺失或模型异常走 `error`。

`safeSend` 解决什么问题：  
统一处理 SSE 发送异常，尤其是连接已关闭时的 `IOException / IllegalStateException`，避免后端直接崩掉。

`AbortController` 解决什么问题：  
让前端能主动停掉当前 AI 请求，并在重复请求或页面离开时中止旧连接。

`requestSeq` 如何防止旧流污染新流：  
每次请求都有本地序号，只有当前序号匹配时才允许继续拼接 `aiAdvice`。

为什么不能说彻底取消上游模型生成：  
因为当前 LangChain4j 这条链路没有给出可取消句柄，只能说“前端停止接收、后端停止继续向关闭连接发送”。

### 5. 部署

Dockerfile 做了什么：  
先构建前端，再打包后端，最终得到可运行的 Spring Boot 镜像。

docker-compose 编排了什么：  
至少编排了 Spring Boot 应用和 MySQL，方便本地或演示环境快速起服务。

Vercel 配置做了什么：  
支持前端单独部署和 API 代理，方便前后端分离联调。

CORS 做了什么：  
后端提供了基础跨域配置，保证前端独立部署时能正常调用 API。

哪些部署表述不能夸大：  
不能说完整安全组配置、不能说生产级 Nginx/监控/高可用、不能说完整线上运维体系都已落地。

## 11. 最终必背 20 问

1. 问题：你介绍一下这个项目。  
30 秒回答：这是一个传统 OJ 判题加 AI 辅助分析的项目。后端先做 Java/C++ 真实编译运行和判题，再把失败上下文结构化传给 AI，最后通过 SSE 流式返回分析建议。  
关键词：判题闭环、结构化上下文、SSE  
证据：`JudgeServiceImpl`、`AiHelpServiceImpl`  
不能说的话：企业级 OJ 平台

2. 问题：你怎么做代码判题？  
30 秒回答：后端根据语言把代码写进临时目录，再走 `javac/java` 或 `g++` 真实编译运行，按测试用例分类返回 Accepted、Compile Error、Runtime Error、Wrong Answer、TLE。  
关键词：临时目录、真实编译、状态分类  
证据：`JudgeServiceImpl`  
不能说的话：异步判题队列

3. 问题：Wrong Answer 怎么判断？  
30 秒回答：程序实际输出和标准输出做标准化后比对，不一致就是 Wrong Answer，同时会返回第一个失败用例的输入、期望输出、实际输出和失败序号。  
关键词：输出比对、失败用例  
证据：`JudgeServiceImpl`、`JudgeResponse`  
不能说的话：返回所有失败用例

4. 问题：Compile Error / Runtime Error 怎么区分？  
30 秒回答：编译阶段失败直接返回 Compile Error；运行阶段退出码非 0 或抛异常时返回 Runtime Error，错误信息放在 `message`。  
关键词：编译阶段、运行阶段、退出码  
证据：`JudgeServiceImpl`  
不能说的话：完整异常堆栈治理体系

5. 问题：为什么只返回第一个失败用例？  
30 秒回答：因为目标是给用户和 AI 最关键的失败样本，同时控制响应体和 Prompt 规模，不是生成完整测试报告。  
关键词：聚焦、控规模  
证据：首次失败即返回  
不能说的话：只是为了偷懒

6. 问题：AI 为什么能分析得更准一点？  
30 秒回答：因为我不是只把错误文本丢给模型，而是把题面、状态、失败输入、期望输出、实际输出和代码结构化传过去。  
关键词：结构化上下文  
证据：`AiHelpRequest`  
不能说的话：模型自己会懂

7. 问题：为什么选 LangChain4j？  
30 秒回答：因为它在 Java 后端里接流式模型比较顺手，特别适合把 DashScope / Qwen 的回调逻辑和服务层整合起来。  
关键词：Java、流式模型  
证据：`pom.xml`、`AiHelpServiceImpl`  
不能说的话：只是为了蹭 AI 热度

8. 问题：Prompt 长度怎么控制？  
30 秒回答：当前做的是字符级长度保护，每个字段有预算，总 Prompt 有上限，代码字段用 head + tail 截断。  
关键词：字段预算、总量预算、head+tail  
证据：`AiHelpServiceImpl`  
不能说的话：严格 2000 Token

9. 问题：为什么不是 2000 Token？  
30 秒回答：因为当前没有 tokenizer，我只能诚实说是字符级控制，而不是 Token 级精确控制。  
关键词：字符级、非 Token 级  
证据：当前代码无 tokenizer  
不能说的话：大概等价

10. 问题：SSE 是怎么做的？  
30 秒回答：后端用 `SseEmitter` 发 `message / done / error`，前端用 `fetch + ReadableStream` 分块读取和解析。  
关键词：SseEmitter、ReadableStream  
证据：`AiHelpServiceImpl`、`App.vue`  
不能说的话：用了 EventSource

11. 问题：为什么不用 EventSource？  
30 秒回答：因为这个 AI 接口需要 POST 请求体，得把题面、代码和失败上下文一起传过去。  
关键词：POST、请求体  
证据：`App.vue`  
不能说的话：不会 EventSource

12. 问题：停止生成怎么实现？  
30 秒回答：前端用 `AbortController` 主动中止请求，后端在连接关闭后停止继续向已关闭连接发送 SSE。  
关键词：AbortController、连接中止  
证据：`stopAiCoach`、`safeSend`  
不能说的话：彻底取消模型生成

30 秒回答：前端给每次 AI 请求分配本地序号，只有当前序号匹配时才允许继续拼接流内容。  
关键词：requestSeq、旧流隔离  
证据：`aiRequestSeq`  
不能说的话：不会串流
